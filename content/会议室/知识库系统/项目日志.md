## 2026-05-25 · Quartz CSS 编译机制研究

### 核心发现

- **base.scss 是死文件**：全源码无一引用，其独特规则（scroll-behavior:smooth、grid-template-areas）不在构建输出中
- 之前关于"base.scss sidebar padding 与 custom.scss 冲突"的调试假设是错的——实际冲突来自组件 SCSS 与 custom.scss

### CSS 编译链（确认）

```
1. Google Fonts 内联 → 2. 组件 SCSS ×16 → 3. custom.scss → 4. joinStyles :root 变量
```

- joinStyles 将 :root CSS 变量追加在所有 SCSS 之后（主题色、字体变量等）
- :root 块会覆盖前面所有对同名 --var 的赋值
- **!important 是唯一能战胜 :root 的手段**（CSS 规范：author !important > author normal，不受源顺序影响）

### 文姬 fonts.scss 方案原理确认

- fonts.scss 通过 @import 放在 custom.scss 末尾
- :root 中 !important 声明 → 战胜 joinStyles 的 :root
- html,body 直接 !important → 双重锁定

### 方法

- 源码全量扫描：components/*.tsx + plugins/*.ts + styles/*.scss，逐一核对 import/@use 引用链
- 构建输出比对：base.scss 独特规则 vs public/index.css，逐一确认存在性
- 验证：构建输出中确实存在两个 :root 块（fonts.scss 带 !important 在前，joinStyles 不带 !important 在后）

### 产出

- content/系统/知识库经验/Quartz-CSS编译机制详解.md（完整分析文档，含编译链全景图、常见坑与对策、推荐工作流）

## 2026-05-26 · 字体修复三回合 + CDN 境外依赖清零

### 字体修复历程（三回合）

- v1 Fraunces+Geist → 中文无变化：纯拉丁字体不含汉字字形
- v2 Noto Serif SC + Noto Sans SC → 线上仍无变化：Google Fonts URL 空格未编码 → 404
- v3 系统原生字体（文姬方案）→ 正文仍无明显变化：bodyFont=微软雅黑=之前回退字体，标题宋体有明显差异

### v3 修复链

- fontOrigin: "googleFonts" → "local"（绕过 GFW 封锁）
- formatFontSpecification: 空格编码为 +（URL bug 修复）
- fonts.scss: Songti SC/SimSun + PingFang SC/Microsoft YaHei 系统字体栈
- 移除 CustomOgImages（local 模式无字体可用）

### 境外 CDN 清零

- Google Fonts CSS + 字体文件 → 清零（fontOrigin: local）
- KaTeX CSS + copy-tex JS → 下载到 quartz/static/，从自有域名加载
- Mermaid ESM → 保留 CDN + bootcdn fallback（含 39 chunk 文件无法本地化）
- cdnjs preconnect → 删除

### 关键经验

1. 字体无变化 ≠ 字体未生效——先检查字体是否含目标语言字形
2. Google Fonts API 200 ≠ 字体能下载——CSS 可访问，字体二进制在 gstatic.com 被墙
3. 多词字体名必须编码空格——formatFontSpecification 直接嵌入，需手动替换 +
4. ESM chunked 库不宜本地化——单文件可行，chunked 成本太高
5. 系统黑体在 Windows = 之前回退字体——正文看不出变化是预期行为
6. base.scss 是死文件——全源码无引用，所有冲突来自组件 SCSS vs custom.scss
