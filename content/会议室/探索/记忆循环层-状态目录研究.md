## 一、OMX 的 .omx/ 设计

OMX 在每个项目下维护 `.omx/` 目录：

```
项目/
├── .omx/
│   ├── state/        ← 当前状态
│   ├── memory/       ← 长期记忆
│   ├── plans/        ← 执行计划
│   └── notes/        ← 会话笔记
```

**关键特性**：Agent 在其中自动读写，实现跨会话状态保持。

---

## 二、我们的等价物

| OMX | Codex 王国 |
|-----|-----------|
| `.omx/state/` | `会议室/计划/` 代办状态 |
| `.omx/memory/` | `会议室/备忘录/` 收拾桌子 |
| `.omx/plans/` | `会议室/计划/` 策略文档 |
| `.omx/notes/` | `会议室/备忘录/` 每日记录 |

### 差异

- OMX 的 `.omx/` 是**每个项目独立**的
- 我们的会议室是**全局共享**的
- OMX 的 agent 可以**自动写入**状态文件
- 我们的写入依赖**玄策手动执行**

### 问题 2：自动写入 vs 手动写入

Codex 没有 OMX 的"agent 自动写入状态"机制。

**替代方案**：依赖 AGENTS.md 的 §0 启动惯例 + §7 收拾桌子，由玄策在每个会话结束时手动记录。

### 问题 3：记忆→规则的反馈回路

目前：备忘录 → 手动提炼 → 更新宪法。

**改进方向**：
- 当备忘录中出现同类型经验 3 次 → 自动提醒提炼
- 每次宪法更新时，注明来源备忘录
