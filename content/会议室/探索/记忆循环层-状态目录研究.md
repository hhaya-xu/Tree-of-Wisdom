## 一、OMX 的 .omx/ 设计

OMX 在每个项目下维护 `.omx/` 目录：

```
项目/
├── .omx/
│   ├── state/        ← 当前状态
│   ├── memory/       ← 长期记忆
│   ├── plans/        ← 执行计划
│   └── notes/        ← 会话笔记
```

**关键特性**：Agent 在其中自动读写，实现跨会话状态保持。

## 二、我们的等价物

| OMX | Codex 王国 |
|-----|-----------|
| `.omx/state/` | `会议室/计划/` 代办状态 |
| `.omx/memory/` | `会议室/备忘录/` 收拾桌子 |
| `.omx/plans/` | `会议室/计划/` 策略文档 |
| `.omx/notes/` | `会议室/备忘录/` 每日记录 |

### 差异

- OMX 的 `.omx/` 是**每个项目独立**的
- 我们的会议室是**全局共享**的
- OMX 的 agent 可以**自动写入**状态文件
- 我们的写入依赖**玄策手动执行**

## 六、四角色反馈回路（2026-05-22 新增）

### 适用范围
反馈回路不是玄策独有的。四个角色各自需要：
- 玄策：策略/评估经验 → 提炼 → 更新宪法
- 明远：编码经验 → 提炼 → 更新执行模式
- 严律：审查经验 → 提炼 → 更新审查标准
- 审固：批评记录 → 提炼 → 完善批评维度

### 待解决问题
- 四个回路的经验如何汇总而不混乱
- 跨角色经验（如：审固发现了明远的模式问题，玄策也需要知道）
- 需要汇总索引还是定期同步机制
- 与知识库建设的关系
