七、伯喈回复（2026-05-25 · Quartz CSS 编译机制研究）

任务：主公令研究 Quartz CSS 机制，避免再犯同样错误

完整溯源

全量扫描 quartz/ 目录：

  • 所有 .tsx/.ts/.scss 文件逐一核查 import/@use 引用
  • componentResources.ts 确认编译入口
  • theme.ts 确认 joinStyles() 逻辑
  • 构建输出 public/index.css 逐规则比对

核心发现

  1. base.scss 是死文件——全源码无一 import，其独特规则（scroll-behavior:smooth、grid-template-areas）不在构建输出中
  2. 之前关于”base.scss sidebar padding 与 custom.scss 冲突”的假设是错的——实际冲突来自组件 SCSS（sidebar-zones.scss 等）+ custom.scss 之间的加载顺序
  3. 真实编译链： Google Fonts → 组件 SCSS x16 → custom.scss → joinStyles :root 变量
  4. :root 在最后——joinStyles() 将 CSS 变量追加在所有 SCSS 之后，覆盖前面所有 —var 声明
  5. !important 是唯一挣脱 :root 的手段（CSS 规范：author !important > author normal，不受源顺序影响）
  6. 文姬 fonts.scss 方案原理确认：:root !important 声明 + html,body 双重锁定 = 正确覆盖 joinStyles :root

产出

  • → Quartz-CSS编译机制详解（content/系统/知识库经验/）
  • 项目日志已更新（项目日志 2026-05-25）

对文姬的后续建议

  • 组件 SCSS（sidebar-zones.scss 等）在 custom.scss 之前加载，同选择器 custom.scss 必胜
  • 只有 :root 变量需要 !important（因为 joinStyles 追加 :root 在最后）
  • 非 :root 的普通样式覆盖不需要 !important——custom.scss 本身就在组件 SCSS 之后
